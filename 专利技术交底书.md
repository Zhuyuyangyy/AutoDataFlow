# AutoDataFlow 数据质量治理平台 — 专利技术交底书 v4.0

## 一、现有技术及缺陷

### 1.1 现有技术概况

数据质量治理领域的主流方案包括：

**（1）单一脚本型数据校验**（如Great Expectations、Pydantic）：
- 通过预定义规则（null比例阈值、值域范围、正则表达式）对数据进行校验
- 缺陷：规则静态，无法适应数据分布的时序演变；每次校验需要全量扫描，资源消耗大

**（2）独立监控看板（如Apache Superset、Grafana）**：
- 将数据质量指标可视化，但数据发现、问题根因定位、修复执行需要手工切换工具
- 缺陷：工具孤岛，监控与修复之间存在断层

**（3）规则引擎型数据质量系统**（如Dataplex、企业级MDM平台）：
- 集中管理数据质量规则，但规则依赖人工配置，覆盖率低
- 缺陷：无法自动发现新数据源和Schema变更；规则与业务语义脱节

**（4）基于统计的Schema推断工具**（如Apache Infer Schema）：
- 从数据样本推断数据类型和基本统计特征
- 缺陷：缺乏跨表血缘推理；无法推断字段间的因果依赖关系

### 1.2 现有技术的共性缺陷

**缺陷1：缺乏自治的端到端处理能力。**
现有系统将"发现→清洗→监控→修复"分散在不同工具中，人工协调成本高。当数据源Schema发生变更时，无法自动触发下游处理链的重新配置。

**缺陷2：多数据源环境下缺乏统一的健康度量化标准。**
MySQL/PostgreSQL/SQL Server/MongoDB等异构数据源的质量指标计算方式各异（MySQL的NULL_COUNT vs PostgreSQL的pg_stat_user_tables），难以跨源统一评估。

**缺陷3：Schema变更的被动响应。**
现有系统在Schema变更发生后被动告警，无法预测变更影响范围（如"新增字段X将导致下游表Y的ETL任务失败"）。

**缺陷4：缺乏因果Schema推断能力。**
现有系统只能推断单表内字段的统计特征，无法从数据中发现字段间的因果依赖关系（如"A字段值变化→导致B字段值变化"的因果链路）。

## 二、本发明的技术方案

### 2.1 总体技术架构

本发明提出**多Agent协同的数据质量自治体系**，四个专业化Agent构成闭环：

```
[Schema Agent] → [ETL Agent] → [Observer Agent] → [Viz Agent]
     ↑                                              │
     └──────────── 反馈循环（Schema回退检测） ←────────┘
```

四个Agent的职责定义如下：

**Schema Agent（模式代理）**：
- 职责：自动扫描数据源目录（./input_data），推断数据类型，生成DDL语句，创建SQLite表
- 输入：CSV/JSON/Parquet文件
- 输出：DDL语句、SQLite表、字段血缘日志
- 核心方法：`infer_schema()`、`dtype_to_sql()`、`generate_ddl()`

**ETL Agent（数据清洗代理）**：
- 职责：使用列式计算引擎（Polars）进行数据清洗、Null值处理、异常值检测
- 输入：原始数据、清洗规则
- 输出：清洗后数据、质量报告
- 核心方法：`clean_data()`、`detect_outliers()`、`transform()`

**Observer Agent（监控代理）**：
- 职责：压力测试、慢查询识别、自动建索引建议、Schema变更检测
- 输入：SQLite表结构、查询日志
- 输出：性能报告、索引建议、变更告警
- 核心方法：`run_stress_test()`、`identify_slow_queries()`、`suggest_indexes()`

**Viz Agent（可视化代理）**：
- 职责：生成多维度质量图表和报告、预测分析
- 输入：质量指标序列、历史数据
- 输出：健康度仪表盘、趋势预测、告警通知
- 核心方法：`generate_dashboard()`、`predict_trend()`、`send_alert()`

Agent间通过标准化消息传递协调，任意Agent失败时触发重试机制（指数退避），确保管道韧性。

#### 创新点1：CausalLineageGraph（因果血缘图谱）

本发明提出**CausalLineageGraph**数据结构，用于表示字段间的因果依赖关系，超越传统数据血缘仅记录"谁引用谁"的局限。

**CausalLineageGraph定义**：

}

**CausalEdge（因果边）的完整数据结构**：

```python
@dataclass
class CausalEdge:
    """
    因果边：表示两个Schema元素之间的因果关系

    与普通数据血缘的区别：
      - 数据血缘边：A → B（A列被B列引用）
      - 因果边：A causes B（A的变化会导致B的变化，包含因果机制）
    """
    source_table: str           # 源表名
    source_column: str         # 源列名
    target_table: str          # 目标表名
    target_column: str         # 目标列名
    mechanism: str             # 因果机制类型
    strength: float = 1.0     # 因果强度 (0-1)
    confidence: float = 1.0    # 推断置信度 (0-1)
    delay_steps: int = 1      # 因果传播延迟（步数）
    conditions: List[str]      # 因果成立的条件
    metadata: Dict[str, Any]  # 附加元数据

    def to_dict(self) -> dict:
        return {
            "source": f"{self.source_table}.{self.source_column}",
            "target": f"{self.target_table}.{self.target_column}",
            "mechanism": self.mechanism,
            "strength": round(self.strength, 3),
            "confidence": round(self.confidence, 3),
            "delay_steps": self.delay_steps,
            "conditions": self.conditions,
        }
```

**CausalLineageGraph的构建方法**（对应代码：`CausalGraphBuilder`类）：

1. **扫描数据库Schema**：从SQLite数据库读取所有表和列信息，生成节点
2. **推断外键因果边**：基于列名模式（如`order_id`→`customer_name`）推断外键关系
3. **从ETL配置构建转换边**：解析血缘日志，添加ETL转换因果边
4. **构建质量传播边**：基于质量指标的时间序列相关性，添加质量传播因果边

```python
class CausalGraphBuilder:
    def build_from_db(self, lineage_log: Optional[List[Dict]] = None) -> "CausalGraphBuilder":
        """从数据库Schema构建因果图谱"""
        conn = sqlite3.connect(self.db_path)
        cursor = conn.cursor()

        # 1. 获取所有表
        cursor.execute("SELECT name FROM sqlite_master WHERE type='table' ORDER BY name")
        tables = [row[0] for row in cursor.fetchall()]

        for table in tables:
            # 添加表节点
            self.nodes[table] = SchemaNode(table=table, node_type="table")

            # 2. 获取所有列
            cursor.execute(f"PRAGMA table_info('{table}')")
            columns = cursor.fetchall()

            for col_info in columns:
                col_name = col_info[1]
                col_dtype = col_info[2]
                pk = col_info[5]

                node_id = f"{table}.{col_name}"
                self.nodes[node_id] = SchemaNode(
                    table=table, column=col_name, dtype=col_dtype, is_key=(pk == 1)
                )

                # 3. 推断外键因果边
                self._infer_foreign_key_edges(cursor, table, col_name)

        # 4. 从血缘日志构建ETL转换因果边
        if lineage_log:
            for entry in lineage_log:
                self._add_etl_edge(entry)

        conn.close()
        return self
```

#### 创新点2：Do-Calculus推理引擎

本发明实现**Do-Calculus推理引擎**（`DoCalculusEngine`类），用于预测Schema变更的因果影响。

**Do-Calculus定义**：

Do-Calculus是Judea Pearl提出的因果推理数学框架，包含三条推理规则：

**Rule 1 (Addition/Removal of Evidence)**：
```
P(Y | do(X), Z, W) = P(Y | do(X), W)  如果 Z ⊥ Y | X, W
```

**Rule 2 (Action/Observation Exchange)**：
```
P(Y | do(X), Z, W) = P(Y | X, Z, W)  如果 Z ⊥ X | Y, W 且无后门路径
```

**Rule 3 (Action/Action Exchange)**：
```
P(Y | do(X), do(Z), W) = P(Y | do(X), Z, W)  如果 X ⊥ Z | Y, W
```

**DoOperation算子的数据结构**：

```python
@dataclass
class DoOperation:
    """
    Do算子：表示对Schema的干预操作

    支持的操作类型：
      - do(drop_column(table, column))
      - do(rename_column(table, column, new_name))
      - do(dtype_change(table, column, new_dtype))
      - do(add_column(table, column, dtype))
      - do(truncate_table(table))
    """
    op_type: str                   # 操作类型
    table: str                    # 表名
    column: Optional[str] = None   # 列名
    new_name: Optional[str] = None # 新列名（用于重命名）
    new_dtype: Optional[str] = None  # 新数据类型（用于类型变更）
    new_value: Any = None         # 新值（用于默认值变更）
    description: str = ""         # 操作描述

    def is_destructive(self) -> bool:
        """是否为破坏性变更"""
        return self.op_type in ("drop_column", "truncate_table", "dtype_change")
```

**DoCalculusEngine的核心方法：`predict_impact()`**

```python
def predict_impact(self, do_op: DoOperation) -> List[CausalEffect]:
    """
    核心方法：预测Do操作的因果影响

    使用Do-Calculus Rule 2（action/observation exchange）：
      P(affected_node | do(operation)) = P(affected_node | operation)

    对于每个受影响的节点，计算：
      1. 因果传播概率（沿因果边的传播）
      2. ETL作业失败概率
      3. 质量降级程度
    """
    effects = []
    affected_nodes = self._get_affected_nodes(do_op)

    for node_id in affected_nodes:
        # 计算因果传播路径
        path = self._find_causal_path(do_op, node_id)

        # 计算影响概率
        probability = self._compute_effect_probability(do_op, node_id, path)

        # 计算严重程度
        severity = self._compute_severity(do_op, node_id, probability)

        # 确定受影响的ETL作业
        etl_jobs = self._get_affected_etl_jobs(node_id)

        effect = CausalEffect(
            do_operation=str(do_op),
            affected_node=node_id,
            effect_type=self._classify_effect(node_id),
            probability=probability,
            severity=severity,
            explanation=self._explain_effect(do_op, node_id, path),
            affected_etl_jobs=etl_jobs,
            path=path,
        )
        effects.append(effect)

    # 按概率排序
    effects.sort(key=lambda e: e.probability, reverse=True)
    return effects
```

**因果效应概率计算公式**：

```python
def _compute_effect_probability(
    self,
    do_op: DoOperation,
    target_node: str,
    path: List[str],
) -> float:
    """
    计算因果效应概率 P(effect | do(op))

    使用贝叶斯推理 + 边权重：
      P(effect | do(op)) = ∏_{edge in path} strength(edge) × base_rate

    其中：
      - path_strength = ∏ strength(edge) × confidence(edge) for edge in path
      - op_multiplier 根据操作类型调整：
          drop_column: 0.95（高破坏性）
          truncate_table: 0.99（极高破坏性）
          rename_column: 0.6（较低破坏性）
          dtype_change: 0.8（中等破坏性）
          add_column: 0.1（低破坏性）
    """
    if not path or len(path) == 1:
        # 直接影响
        base = 0.95 if do_op.is_destructive() else 0.7
        return base

    # 计算路径乘积
    path_strength = 1.0
    for i in range(len(path) - 1):
        src, tgt = path[i], path[i + 1]
        edge = self._find_edge(src, tgt)
        if edge:
            path_strength *= edge.strength * edge.confidence
        else:
            path_strength *= 0.8  # 默认衰减

    # 应用操作类型乘子
    op_multiplier = {
        "drop_column": 0.95,
        "truncate_table": 0.99,
        "rename_column": 0.6,
        "dtype_change": 0.8,
        "add_column": 0.1,
    }.get(do_op.op_type, 0.5)

    return min(0.999, path_strength * op_multiplier)
```

**CausalEffect（因果效应）的数据结构**：

```python
@dataclass
class CausalEffect:
    """
    因果效应：描述一次Do操作产生的因果效应
    """
    do_operation: str          # Do操作描述
    affected_node: str          # 受影响节点
    effect_type: str           # 效应类型："quality_degradation", "etl_failure", "cascading"
    probability: float          # 发生概率 P(effect | do(op))
    severity: str              # 严重程度："info", "warning", "critical"

    path: List[str]            # 因果传播路径
```

#### 创新点3：CounterfactualReasoner（反事实推理引擎）

本发明实现**反事实推理引擎**（`CounterfactualReasoner`类），用于回答"What-if"类型的问题。

**反事实推理的定义**：

反事实推理回答"如果当初...会怎样"的问题。在Schema变更场景中：
- **事实结果（Factual）**：不进行干预时的结果
- **反事实结果（Counterfactual）**：进行干预后的结果
- **差异（Difference）**：两者之间的区别

**CounterfactualResult的数据结构**：

```python
@dataclass
class CounterfactualResult:
    """
    反事实推理结果
    """
    question: str               # 反事实问题
    factual: str               # 事实结果（未干预时的结果）
    counterfactual: str         # 反事实结果（干预后的结果）
    difference: str            # 差异描述
    confidence: float          # 推理置信度
    causal_path: List[str]      # 因果路径
    mitigation_suggestions: List[str]  # 缓解建议
```

**CounterfactualReasoner的核心方法：`reason()`**

```python
def reason(
    self,
    change: Dict,              # 变更描述，如 {"type": "drop_column", "table": "orders", "column": "customer_email"}
    outcome: str,              # 预测结果，如 "ETL_Job_1", "quality_score", "revenue_total"
    counterfactual_question: Optional[str] = None,
) -> CounterfactualResult:
    """
    执行反事实推理

    参数：
        change: Dict，描述变更
        outcome: str，要预测的结果指标
        counterfactual_question: str，可选的自然语言问题

    返回：
        CounterfactualResult，包含事实结果和反事实结果
    """
    # 构建DoOperation
    do_op = DoOperation(
        op_type=change["type"],
        table=change["table"],
        column=change.get("column"),
        new_name=change.get("new_name"),
        new_dtype=change.get("new_dtype"),
    )

    # 预测干预后的结果
    effects = self.do_calculus.predict_impact(do_op)

    # 查找与outcome相关的效应
    relevant_effects = self._filter_effects_by_outcome(effects, outcome)

    # 计算反事实结果
    counterfactual_value = self._compute_counterfactual(do_op, relevant_effects, outcome)

    # 事实结果（无干预）
    factual_value = self._get_factual_value(outcome)

    # 差异
    difference = self._compute_difference(factual_value, counterfactual_value, outcome)

    # 生成缓解建议
    suggestions = self._generate_mitigation(do_op, relevant_effects)

    return CounterfactualResult(
        question=counterfactual_question or f"What if we {change['type']} {change.get('table', '')}.{change.get('column', '')}?",
        factual=factual_value,
        counterfactual=counterfactual_value,
        difference=difference,
        confidence=self._compute_confidence(relevant_effects),
        causal_path=self._get_causal_path_summary(do_op, outcome),
        mitigation_suggestions=suggestions,
    )
```

**反事实推理的应用场景**：

| 场景 | 问题 | 回答 |
|------|------|------|
| 删除字段 | "如果我们删除orders.customer_email，哪些ETL作业会失败？" | "预计2个ETL作业受影响：ETL_JoinMasterData, ETL_EmailValidation" |
| 重命名字段 | "如果我们把sales.amount重命名为sales.revenue，质量评分会下降多少？" | "质量评分预计下降12.3分（概率78%）" |

**缓解建议生成示例**：

```python
def _generate_mitigation(
    self,
    do_op: DoOperation,
    effects: List[CausalEffect],
) -> List[str]:
    """生成缓解建议"""
    suggestions = []

    if do_op.op_type == "drop_column":
        suggestions.append(f"在删除 {do_op.table}.{do_op.column} 前，创建视图或保留列作为过渡")
        suggestions.append("检查所有引用该列的ETL作业，优先更新下游依赖")
        suggestions.append("使用影子表（shadow table）验证变更安全性")
    elif do_op.op_type == "rename_column":
        suggestions.append(f"使用数据库重命名（ALTER TABLE ... RENAME COLUMN）而非应用层修改")
        suggestions.append("添加注释说明列名变更，保持向后兼容")
    elif do_op.op_type == "dtype_change":
        suggestions.append("在变更前验证目标类型是否兼容现有数据")
        suggestions.append("使用CAST函数在ETL层做类型转换验证")

    return suggestions
```

#### 创新点4：SchemaCausalityMetrics（因果Schema度量指标）

本发明定义**SchemaCausalityMetrics**用于量化因果Schema的质量和完整性。

**血缘密度（Lineage Density）**：

```python
def lineage_density(graph: CausalLineageGraph) -> float:
    """
    血缘密度 = 因果边数量 / 最大可能边数量

    公式：
        LD = |E| / (|V| × (|V| - 1))

    其中：
        |E| = 因果边数量
        |V| = 节点数量

    用途：衡量Schema的因果关联程度
        - LD < 0.1：稀疏因果，字段间独立性高
        - LD > 0.5：稠密因果，字段间高度关联
    """
    n = len(graph.nodes)
    if n <= 1:
        return 0.0
    max_edges = n * (n - 1)
    return len(graph.edges) / max_edges
```

**传播效率（Propagation Efficiency）**：

```python
def propagation_efficiency(graph: CausalLineageGraph, source: str, target: str) -> float:
    """
    传播效率 = 直接因果强度 / 路径长度

    公式：
        PE(s→t) = strength(s→t) / length(s→t)

    其中：
        strength(s→t) = 沿最短因果路径的累积因果强度
        length(s→t) = 最短因果路径的边数

    用途：衡量因果信息从源到目标的传播效率
        - PE > 0.8：高效传播，变更影响快速传导
        - PE < 0.3：低效传播，变更影响衰减严重
    """
    path = shortest_causal_path(graph, source, target)
    if not path:
        return 0.0

    path_strength = 1.0
    for i in range(len(path) - 1):
        edge = find_edge(graph, path[i], path[i + 1])
        if edge:
            path_strength *= edge.strength

    return path_strength / len(path)
```

**脆弱性评分（Vulnerability Score）**：

```python
def vulnerability_score(graph: CausalLineageGraph, node: str) -> float:
    """
    脆弱性评分 = 受影响下游节点数 × 平均影响概率

    公式：
        VS(n) = |Downstream(n)| × Avg(P(effect | do(n)))

    其中：
        |Downstream(n)| = 节点n的所有下游节点数量
        Avg(P(effect | do(n))) = 平均因果效应概率

    用途：识别高风险节点
        - VS > 50：高脆弱性，变更将导致大量下游受影响
        - VS < 10：低脆弱性，变更影响有限
    """
    downstream = graph.get_downstream(node)
    if not downstream:
        return 0.0

    total_prob = 0.0
    for tgt in downstream:
        path = graph.find_path(node, tgt)
        prob = compute_effect_probability(path)
        total_prob += prob

    avg_prob = total_prob / len(downstream)
    return len(downstream) * avg_prob
```

**质量传播系数（Quality Propagation Coefficient）**：

```python
def quality_propagation_coefficient(graph: CausalLineageGraph) -> Dict[str, float]:
    """
    质量传播系数：描述质量问题的传播能力

    公式：
        QPC(n) = Σ_{e outgoing from n} strength(e) × quality_degradation(e)

    返回：
        Dict[node_id, QPC]，每个节点的传播能力

    用途：
        - 识别"质量瓶颈"节点：QPC最高的节点是质量问题的放大器
        - 优先监控高QPC节点可以最大化质量治理效率
    """
    qpc = {}
    for node_id in graph.nodes:
        outgoing_edges = [e for e in graph.edges
                         if f"{e.source_table}.{e.source_column}" == node_id]
        total_qpc = sum(e.strength * (1 - e.target_node.quality_score / 100)
                       for e in outgoing_edges)
        qpc[node_id] = total_qpc
    return qpc
```

**重试机制（指数退避）**：

```python
def execute_with_retry(
    agent: Agent,
    message: AgentMessage,
    max_retries: int = 3,
    base_delay: float = 1.0,
) -> ExecutionResult:
    """
    带指数退避的重试机制

    公式：
        delay = base_delay × 2^retry_count + random_jitter

    其中：
        retry_count = 已重试次数
        random_jitter = uniform(0, 0.1 × base_delay)

    示例：
        第1次重试：delay = 1.0 × 2^0 + jitter = 1.0s
        第2次重试：delay = 1.0 × 2^1 + jitter = 2.0s
        第3次重试：delay = 1.0 × 2^2 + jitter = 4.0s
    """
    for attempt in range(max_retries):
        try:
            result = agent.process(message)
            return ExecutionResult(success=True, result=result)
        except AgentExecutionError as e:
            if attempt == max_retries - 1:
                raise MaxRetriesExceeded(f"Agent {agent.name} failed after {max_retries} attempts")

            delay = base_delay * (2 ** attempt) + random.uniform(0, 0.1 * base_delay)
            time.sleep(delay)

            message.retry_count = attempt + 1
            logger.warning(f"Retry {attempt + 1}/{max_retries} for {agent.name}, delay={delay:.2f}s")
```

## 五、权利要求书

### 独立权利要求

**权利要求1**（多Agent协同数据质量自治系统）：
一种基于多Agent协同的数据质量自治系统，其特征在于，包括：
- Schema Agent：自动扫描数据源目录，推断数据类型，生成DDL并创建SQLite表，构建ODS→DWD→ADS三层数仓
- ETL Agent：使用列式计算引擎进行数据清洗和异常值检测，构建字段级血缘日志
- Observer Agent：压力测试、慢查询识别、自动建索引建议、Schema变更检测
- Viz Agent：生成多维度质量图表、健康度评分和趋势预测报告
- 各Agent间通过标准化AgentMessage消息传递协调，构成端到端闭环
- 任意Agent失败时触发指数退避重试机制，确保管道韧性

**权利要求2**（基于CausalLineageGraph的因果血缘表示）：
一种基于因果血缘图谱的数据质量建模方法，其特征在于，构建CausalLineageGraph数据结构，包括：
- 节点集合：SchemaNode表示表或列，包含table、column、dtype、quality_score、is_key属性
- 边集合：CausalEdge表示因果关系，包含source_table、source_column、target_table、target_column、mechanism、strength、confidence、delay_steps属性
- 因果机制类型：CausalMechanism枚举，定义FOREIGN_KEY、ETL_TRANSFORM、SCHEMA_DEPENDENCY、QUALITY_PROPAGATION、CASCADING_FAILURE五种机制
- 所述CausalEdge与传统数据血缘的区别在于：包含因果机制类型、因果强度、推断置信度、传播延迟参数

**权利要求3**（Do-Calculus推理引擎）：
一种基于Do-Calculus的Schema变更影响预测方法，其特征在于，包括步骤：
- 定义DoOperation算子表示对Schema的干预操作，支持drop_column、rename_column、dtype_change、add_column、truncate_table类型
- 使用Do-Calculus Rule 2（action/observation exchange）：P(affected_node | do(operation)) = P(affected_node | operation)
- 对每个受影响节点，计算因果传播概率：P(effect | do(op)) = ∏_{edge in path} strength(edge) × base_rate
- 根据操作类型应用乘子：drop_column(0.95)、rename_column(0.6)、dtype_change(0.8)、add_column(0.1)
- 输出CausalEffect列表，包含affected_node、probability、severity、explanation、affected_etl_jobs、path属性

**权利要求4**（反事实推理引擎）：
一种基于反事实推理的质量影响分析方法，其特征在于，包括步骤：
- 构建CounterfactualResult数据结构，包含question、factual、counterfactual、difference、confidence、causal_path、mitigation_suggestions属性
- 接收change参数描述Schema变更，outcome参数指定要预测的结果指标
- 计算factual结果（无干预时的结果）和counterfactual结果（干预后的结果）
- 生成缓解建议列表，包含操作前准备、操作中监控、操作后验证三类建议
- 支持What-if问题回答，如"如果我们删除orders.customer_email，哪些ETL作业会失败？"

**权利要求5**（SchemaCausalityMetrics量化指标）：
一种基于因果Schema的质量度量方法，其特征在于，定义以下指标：
- 血缘密度（LineageDensity）：LD = |E| / (|V| × (|V| - 1))，衡量Schema的因果关联程度
- 传播效率（PropagationEfficiency）：PE = strength(s→t) / length(s→t)，衡量因果信息从源到目标的传播效率
- 脆弱性评分（VulnerabilityScore）：VS(n) = |Downstream(n)| × Avg(P(effect | do(n)))，识别高风险节点
- 质量传播系数（QualityPropagationCoefficient）：QPC(n) = Σ strength(e) × quality_degradation(e)，识别质量瓶颈节点

### 从属权利要求

**权利要求6**（如权利要求1所述的系统）：
所述各Agent通过标准化消息格式协调，消息格式包括sender、receiver、msg_type、payload、timestamp、correlation_id、retry_count字段，支持SCHEMA_DISCOVERED、DATA_READY、ETL_COMPLETED、QUALITY_REPORT、ALERT_TRIGGERED、SCHEMA_CHANGE_DETECTED六种消息类型。

**权利要求7**（如权利要求1所述的系统）：
所述重试机制使用指数退避公式：delay = base_delay × 2^retry_count + random_jitter，其中random_jitter = uniform(0, 0.1 × base_delay)，max_retries默认为3次。

**权利要求8**（如权利要求2所述的方法）：
所述CausalEdge的mechanism属性为FOREIGN_KEY时，strength默认为1.0，confidence基于列名模式匹配度计算（如_id→_name匹配）；为ETL_TRANSFORM时，strength从血缘日志读取。

**权利要求9**（如权利要求3所述的方法）：
所述predict_impact方法对每个CausalEffect计算severity严重程度：若target_node为is_key或probability>0.8则为critical；若probability>0.5则为warning；否则为info。

**权利要求10**（如权利要求3所述的方法）：
所述CausalEffect的effect_type属性包括：quality_degradation（质量降级）、etl_failure（ETL作业失败）、cascading（连锁故障）、metric_error（指标计算错误）。

**权利要求11**（如权利要求4所述的方法）：
所述counterfactual方法支持三种outcome类型：quality_score（质量评分）、etl_job（ETL作业）、revenue_total（指标计算），根据类型计算不同的反事实结果格式。

**权利要求12**（如权利要求4所述的方法）：
所述生成缓解建议包括：
- 对于drop_column操作：建议创建保留视图、使用影子表验证、优先更新下游依赖
- 对于rename_column操作：建议使用ALTER TABLE RENAME COLUMN原子操作、添加列注释保持向后兼容
- 对于dtype_change操作：建议先在ETL层添加类型转换验证、检查目标类型范围覆盖现有数据

**权利要求13**（如权利要求5所述的方法）：
所述血缘密度LD用于判断Schema特征：LD<0.1为稀疏因果（字段间独立性高），LD>0.5为稠密因果（字段间高度关联）。

**权利要求14**（如权利要求5所述的方法）：
所述脆弱性评分VS>50的节点被标记为高脆弱性，变更将导致大量下游受影响；VS<10为低脆弱性，变更影响有限。

**权利要求15**（如权利要求5所述的方法）：
所述质量传播系数QPC用于识别质量瓶颈节点，QPC最高的节点是质量问题的放大器，优先监控高QPC节点可以最大化质量治理效率。

**权利要求16**（一种计算机可读存储介质）：
一种计算机可读存储介质，其特征在于，存储程序代码，当所述程序代码被处理器执行时，使处理器执行权利要求1至15任一项所述的方法步骤。
